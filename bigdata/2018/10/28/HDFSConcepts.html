HDFS Concepts

October-28-2018

Blocks




blocksize: The minimum amount of data that a disk can read/write. This is usually hidden from the user.

  • file system: few kilobytes.
  • disk: 512 bytes.

In HDFS,

  • The default block size is 128 MB. This large size is to minimize the cost of seeks.
  • Files(large) in HDFS are broken into blocks and stored as independent units.
  • If a file is smaller than a single block. Unlike file systems it doesnt occupy a full block of undelying storage.

This block abstraction has several advantages:

  • A file can be larger than any single disk in the distributed system
  • It simplifies the storage subsystem as,
    • Its easy to calculate how many blocks can be stored at a single node.
    • Eliminates metadata concern(as blocks are simple chunks of data are not a file anymore).
  • Blocks can be replicated (default value is 3) to make the system available and fault tolerant.

Name Nodes and Data Nodes

HDFS cluster is made up of two types of nodes which work in a master worker fashion. They are:

  • Name node
  • Data node

The client is presented with a file system interface to interact with HDFS so the functioning of name node and data node is transparent.

Name Node:

  • manages the file system namespace
  • maintains file system tree and meta data for all the files and directories
  • The above information is stored persistantly on the disk as two files:
    • Namespace image
    • edit log
  • Also has infomation about rhe locations of data nodes on which a file’s blocks are stored.

Datanode:

  • workhorses of the system.
  • store & retrive blocks when told to
  • Periodically report to namenode what blocks they are holding.

Without name node HDFS can’t be used. Hence it is important to make namenode resilient to failure. Hadoop supports two ways of doing this:

  • Backup files that make up the persistant state of the systems metadata(back up metadata):
    • hadoop can be configured to write persisant state infomation to multiple Files systems i.e to local disk as well as a Network Files system mount(attached).
    • These writes need to be synchronous and atomic
  • Have a Secondary Name Node:
    • despite of the name it doesn’t act as a name node.
    • main role: Periodically merge the namespace image with edit log to prevent it from becoming too large and keep a copy of the merged image.
    • It runs on a seperate physical machine.
    • The state of SNN lags from that of primary. So incase of NN failure data loss is certain.

Hence, if the NN fails we copy the metadata files on NFS to SNN and run it as primary.

HDFS Federation

Name node stores reference to every file’s blocks in the file system. So, if a file system has large number of files the memory becomes and issue. #federationTotheRescue:

  • Federation allows clustes to scale by adding name nodes to manage a portion of the file system.
  • Each name node manages a seperate namespace volume. Which is made up of:
    • namespace metadata
    • block pool: blocks for each files
  • Ex: one name node can manage the files and directories under /user, the other under /share etc.
  • NSV are independent of each other:
    • failure of one doesnt affect each other.
  • However the block pools are not partioned i.e data nodes store blocks from multiple federations.

High Availability




Name node is the SPOF(single point of failure) of HDFS. If a name node fails:

  • All clients can’t read, write or list files as it is the sole repository of metadata
  • Hence, whole HDFS is out of service.

So if a name node fails, To recover:

  • Admin starts a new name node with one of the file system metadata replicas. However this new name node wont serve until:
    • It has loaded namespace image into memory.
    • replayed the edit log.
    • receives enough block reports from data nodes to leave safe mode.
  • And configures the data nodes and clients to use this name node.

The above recovery process for a sufficiently large cluster takes atleast 30min. Hadoop solved this problem by adding support for High Availability. However this requires a few changes:

  • There are a pair of name nodes:
    • one active
    • one standby: if active fails it takes over without a signficant delay.
  • These name nodes must use highly available shared storage
  • Data nodes must send block reports to both name nodes
  • Client must be configured to handle failure.

There are two choices for highly available shared storage:

  • NFS filer
  • Quorum Journal Manager(QJM):
    • runs a group of journal nodes. Typically 3
    • each edit must be written to a majority of journal nodes.

Failover and Fencing

The transistion from active to stand by name nodes is manageb by an entity known as failover controller. default is ZooKeeper Implementation(ensure only one name node is active).

Each name node runs a light weight failover controller process whose job is to monitor and trigger a failover if it fails.

This failure can be issued manually in case of maintainence. This is known as graceful failure. Since failover controller arranges transistion for name nodes to switch roles.




Incase, of an ungraceful failover:

  • It is impossible to know whether failed namenode has stopped running.
  • “The process of preventing the previously active namenode from casuing corruption” - #fencing. Some fencing mechanisms are :
  • Revoking name node access to shared storage.
  • Disabling its network port.
  • STONITH: “Shoot the other node in the Head”. Forcibly shutdown the host machine.
